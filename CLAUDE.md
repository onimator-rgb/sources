# Onimator — Project Overview

## What this software does
Onimator is a multi-device automation platform for social apps, including Instagram, TikTok, dating apps, and similar platforms. It coordinates and runs automation tasks across many device-bound accounts at scale. The system is designed around centralized orchestration, per-device automation execution, database-backed state management, configuration-driven behavior, and runtime logging.

## Business context
Onimator supports commercial services offered through:
- `wizzysocial.com`
- `insta-max.pl`
- `wypromujemy.com`

This is production-oriented software. Changes should be evaluated in terms of reliability, account performance visibility, scalability, and operator usefulness.

## Primary goals
- Run automation tasks reliably across many devices.
- Support multi-account, multi-device operation at scale.
- Keep per-device execution isolated and recoverable.
- Maintain strong database performance as the device count grows.
- Keep memory usage controlled for long-running instances.
- Capture enough logs and state to debug failures quickly.
- Improve observability of account performance and operational issues.

## Strategic AI / reporting goal
One important product goal is to build an agent that analyzes files generated by the bot and turns them into useful reports.

That agent should:
- read bot-generated files, logs, and exports
- analyze account-level performance
- generate reports that are understandable by operators
- highlight anomalies, weak points, and trends
- explain what the team should look at first
- help separate signal from noise in large volumes of runtime data

## What the reporting agent should help answer
- Which accounts are performing well, and which are underperforming?
- Which trends are improving or declining over time?
- Which accounts need attention first?
- Which recurring errors or warnings are most important?
- Are problems caused by device instability, configuration issues, retry loops, data issues, or platform-side limits?
- What should operators inspect next to improve account performance?

## Likely input sources for the reporting agent
- `/logs` for runtime logs and diagnostic traces
- `/database` for structured state, exports, or snapshots
- `/devices` for per-device outputs and automation artifacts
- `/config` for thresholds, flags, and runtime settings needed to interpret behavior
- any bot-generated CSV, JSON, TXT, or report files produced during runs

## High-value signals to surface in reports
Depending on what the bot files contain, prioritize surfacing:
- account-level performance trends
- success versus failure rates of actions
- retry spikes and recurring error patterns
- device-level instability or slowdown
- abnormal drops, inconsistent results, or outlier accounts
- differences between accounts, device groups, or time windows
- missing data versus true zero-performance cases

- `/src` — Main application code.
- `/database` — Database files, schemas, migrations, and query logic.
- `/logs` — Runtime logs and diagnostic output.
- `/config` — Configuration files and environment-specific settings.

## Likely system responsibilities
- Orchestrate automation jobs across many devices.
- Manage account and device state over long-running sessions.
- Persist execution state and results in the database.
- Load runtime configuration from files or environment settings.
- Record logs for auditing, debugging, retries, and recovery.
- Handle failures, retries, and partial-success scenarios without destabilizing the full system.
- Support reporting and analysis workflows based on bot-generated files.

## Performance priorities
1. Speed of automation tasks per device.
2. Database query performance at scale, especially with 100+ devices.
3. Memory usage per running instance.
4. Error rate and retry logic quality.
5. Stability of long-running automation workers.
6. Quality and usefulness of performance reporting.

## Folder-level notes
### `/src`
This should contain the main orchestration and business logic. Likely areas include task coordination, worker lifecycle management, shared services, utilities, and any API or internal control flows.

### `/database`
This should contain schemas, migrations, seed data, query helpers, repository logic, or ORM configuration. Treat the database as an important source of truth for runtime state and recovery.

### `/logs`
This stores runtime logs and diagnostic traces. These logs are important for debugging device-level failures, understanding retries, and feeding future reporting or analysis agents.

### `/config`
This should hold application configuration, feature flags, environment-specific settings, and other runtime options. Review config loading paths before changing behavior that differs by environment.

### `/devices`
This should contain device-specific automation logic. This area is likely latency-sensitive and failure-prone because it interacts with external apps and device environments. Changes here should be conservative and observable.

## Engineering expectations
- Prefer small, behavior-preserving changes over broad refactors.
- Treat device workflows as flaky external systems that require retries and recovery.
- Keep per-device logic isolated so one failing device does not impact the whole fleet.
- Make state transitions explicit when possible.
- Keep logs structured and useful for tracing failures.
- Be careful with concurrency, long-running workers, and shared resources.
- Avoid changes that significantly increase per-device memory or database load.
- When possible, preserve or improve the quality of data that downstream reporting depends on.

## Change guidelines for Claude Code
When modifying this codebase:
- First identify where device task scheduling, execution, retry, and completion are handled.
- Check how runtime worker state maps to database state.
- Review config loading before assuming defaults.
- Preserve existing retry and recovery behavior unless intentionally changing it.
- When debugging, use both logs and database records to trace failures.
- When optimizing performance, prioritize per-device throughput, query efficiency, and memory footprint.
- If a change touches `/devices`, consider timeout handling, retries, logging, and state persistence together.
- If a change affects generated files, logs, or exports, consider downstream reporting impact.
- If building the reporting agent, normalize raw bot files into a stable schema before summarizing results.
- Reports should distinguish facts, assumptions, anomalies, and recommended next checks.
- If the existing structure differs from this document, update this file to reflect the real architecture.

## Common areas to inspect first
- Job orchestration and worker lifecycle.
- Device session handling and automation entry points.
- Database read/write paths related to task state.
- Logging and error-handling middleware.
- Configuration bootstrap and environment overrides.
- Any code that generates files later consumed for analysis or reporting.

## Reporting-agent implementation notes
When building the account-performance analysis agent:
- expect inconsistent or noisy bot-generated files
- make parsing resilient and explicit
- map raw outputs to account, device, timestamp, and result type where possible
- prefer reproducible summaries over ad hoc interpretation
- rank issues by likely impact so operators know what to inspect first
- clearly separate missing data, parser errors, and true account-performance problems
- keep generated reports concise, actionable, and traceable back to source files
